STEP 01 GitHub って何?なぜゲーム保管に最適?

GitHub は 世界最大のコード倉庫

一言で言うと:あなたが作ったコード(ゲーム)をクラウドに無料で保管できるサービス。世界中の開発者・1.5 億人以上が使う、業界の標準ツールです。

🎯 ゲーム保管に GitHub を選ぶ 5 つの理由

  1. 無料で使える - Public(公開)リポジトリは無料・容量制限ほぼなし
  2. パソコンが壊れても安心 - クラウドに保管されるのでデータ消失なし
  3. 変更履歴が全部残る - 「あの時のバージョンに戻したい」が一発でできる
  4. GitHub Pages で即公開 - URL を友達に送るだけでブラウザでプレイしてもらえる
  5. Claude Code との相性◎ - Claude Code から直接アップロード可能

🆚 Git と GitHub の違い(混乱しがち)

  • Git(ぎっと):変更履歴を管理するソフト。自分のパソコン上で動く
  • GitHub(ぎっとはぶ):Git で管理されたファイルを クラウドに置く サービス

💡 例えるなら:Git は「セーブ機能」、GitHub は「クラウドストレージ」。最初は GitHub だけ使えれば OK です。

💡 こんな場面で活躍します 「PCが壊れた!」→ 大丈夫、GitHub にある / 「友達にゲームを遊ばせたい」→ URL 送るだけ / 「3 日前の状態に戻したい」→ コミット履歴から復元 / 「友達と一緒に作りたい」→ コラボ機能で共同開発
02
STEP 02 アカウント作成(無料・3 分)
›

無料で すぐ始められます

クレジットカード不要、メールアドレス 1 つで登録完了。

① 公式サイトへアクセス

github.com/signup を開きます。

② メールアドレス・パスワード・ユーザー名

  • メールアドレス:普段使うもの(後で確認メールが届きます)
  • パスワード:15 文字以上 or 8 文字以上(数字・小文字を含む)
  • ユーザー名:あなたの GitHub URL の一部になります
⚠️ ユーザー名は重要 あなたの GitHub URL は github.com/ユーザー名 になり、ゲーム公開時の URL も ユーザー名.github.io/repo名 という形式に。短く・覚えやすく・本名 or プロ名で決めるのがおすすめ。後で変更可能ですが、URL が変わるので最初によく考えて。

③ 認証コードを入力

パズル認証 → メールに届いた 8 桁のコードを入力 → ログイン完了。

④ プランを選ぶ

「Free」プランを選択。これで全機能の 95% が使えます。

✅ こうなれば成功 ログイン後、右上にあなたのアイコンが表示され、ダッシュボード画面が見えれば登録完了。さっそくゲーム保管を始められます。
🔐 2FA(2 段階認証)を強く推奨 設定 → Password and authentication → Two-factor authentication で有効化。スマホアプリ(Google Authenticator など)を使えば、不正ログインを防げます。2024 年から GitHub では 2FA が必須化されているため、いずれ設定が必要になります。
03
STEP 03 専門用語をやさしく(コミット・プッシュ・etc)
›

用語を 身近なもので例えます

これだけ覚えれば 90% は理解できます。

📁 リポジトリ(Repository / Repo)

= ゲームを入れるフォルダ。1 つのゲームに 1 つのリポジトリを作るのが基本。

例:「テトリス」「シューティング」「パズルゲーム」と 3 つのゲームがあるなら、リポジトリも 3 つ。

📸 コミット(Commit)

= 「ここまで保存!」のスナップショット。ゲームでいうセーブポイントのようなもの。

例:「キャラ追加完了」「敵の動きを修正」のように小さく区切ってコミットすると、後で履歴を辿りやすい。

⬆️ プッシュ(Push)

= 自分のパソコンから GitHub に送信。「アップロード」と思っておけば OK。

⬇️ プル(Pull)

= GitHub から自分のパソコンに取得。「ダウンロード」と思っておけば OK。

💡 別の PC で続きを作るとき、または友達と共同作業するときに使います。

🌳 ブランチ(Branch)

= 並行作業の枝。最初は気にしなくて OK。1 人でゲームを作る間は main ブランチだけ使えば十分です。

💡 慣れてきたら「実験的な機能を試す用のブランチ」を切ると、本体を壊さず冒険できます。

📄 README(リードミー)

= リポジトリの表紙ページ。ゲームの説明文を README.md という名前のファイルで書きます。Markdown 形式で書け、リポジトリのトップに自動表示されます。

💡 全部覚えなくていい 最初は「リポジトリ = フォルダ」「コミット = 保存」「プッシュ = アップロード」だけ覚えて始めてOK。残りは使いながら自然に身につきます。

04
STEP 04 最初のリポジトリを作る
›

ゲーム用の 箱を作る

1 つのゲームに 1 つのリポジトリ。所要時間 1 分。

① 「+」ボタン → New repository

GitHub 画面右上の 「+」 アイコンをクリック → 「New repository」 を選択。

② リポジトリ名を入力

  • 英数字・ハイフン(-)・アンダースコア(_)が使える
  • 日本語・スペースは NG
  • 例:tetris-game / my-shooter / memory-puzzle

💡 後で公開するときの URL の一部になります。短くてわかりやすい名前を。

③ Description(説明・任意)

「2 分で遊べるシューティングゲーム」のように一言で書く。空欄でも OK。

④ Public か Private を選ぶ

  • 🌍 Public(公開):誰でも見られる。GitHub Pages で公開するなら必須
  • 🔒 Private(非公開):自分だけ見られる(チームメンバーは招待で追加可能)
💡 ゲーム保管+公開なら Public 一択 「コードを読まれるのが嫌」なら Private にできますが、その場合 GitHub Pages 公開には Pro プラン($4/月)が必要。最初は Public で問題なし。コードがマネされる心配は実質ありません。

⑤ 「Add a README file」にチェック

これにチェックを入れると、リポジトリ作成時に README.md が自動生成されます。あとで編集できるので必ずチェック ON で。

⑥ 「Create repository」をクリック

これでリポジトリ完成!ゲームを置く準備ができました。

✅ こうなれば成功 github.com/ユーザー名/リポジトリ名 というページが表示され、README.md ファイルが 1 つだけ置いてあれば OK。
05
STEP 05 ゲームファイルをアップロード
›

ブラウザから ドラッグ&ドロップ

コマンド不要・GUI だけで完了する一番ラクな方法。

① 「Add file」→「Upload files」

リポジトリ画面で、緑の「Add file」ボタン → 「Upload files」をクリック。

② ファイルをドラッグ&ドロップ

Claude Code で作ったゲームのファイル(index.html、style.css、script.js、画像フォルダなど)をまるごと枠内にドロップ。

💡 フォルダごとアップロード可。ただし1 ファイル 25MB 以下、合計 100MB 以下が目安。

③ コミットメッセージを入力

下のフォームに「初めてのアップロード」など好きなメッセージを記入。空欄だと自動で「Add files via upload」になります。

④ 「Commit changes」で確定

緑のボタンをクリック → 数秒待つ → ファイルがリポジトリに反映されます。

✅ こうなれば成功 リポジトリ画面で index.html、style.css、script.js などのファイルが一覧表示されていれば成功。これでゲームが永久保管されました 🎉

📁 ゲームの推奨フォルダ構造

my-game/
├─ index.html      ← ★ 必須(GitHub Pages のトップページ)
├─ style.css       ← デザイン
├─ script.js       ← ゲームロジック
├─ assets/         ← 画像・音声フォルダ
│  ├─ player.png
│  └─ bgm.mp3
└─ README.md       ← ゲームの説明書

💡 index.html という名前のファイルが ルート(一番上)に必ず必要。これが GitHub Pages の入り口になります。

⚠️ 機密情報を含めない API キー・パスワード・メアドなどを含むファイルを Public リポジトリにアップロードすると 世界中に公開されます。.env などの設定ファイルは絶対にアップしない。間違えてアップしたら、パスワードや API キーは即座にローテーション(再発行)を。
06
STEP 06 GitHub Pages で公開する 🎉
›

友達にゲームを送れる魔法のスイッチ

このステップが GitHub の最大の魅力。3 分で全世界からあなたのゲームにアクセスできるようになります。

① リポジトリの「Settings」へ

リポジトリ画面の上メニュー、一番右にある 歯車アイコンの「Settings」 をクリック。

② 左メニューから「Pages」

左サイドバーの中ほどに「Pages」項目があるのでクリック。

③ Source の設定

  • 「Source」を 「Deploy from a branch」に
  • 「Branch」を 「main」、フォルダを 「/(root)」に
  • 「Save」をクリック

④ 数分待つ

Save した後、1〜10 分でデプロイ(公開)が完了します。ページを再読み込みすると、上部に 緑色の枠で URL が表示されます。

⏰ 初回は反映に時間がかかることが多いので、慌てず待ちましょう。

⑤ URL でアクセス!

表示された URL は次のような形式です:

https://ユーザー名.github.io/リポジトリ名/

クリックして、あなたのゲームがブラウザで動けば 大成功 🎉

✅ ここまで来たらあなたは「インディーゲーム公開者」 この URL を友達・SNS・履歴書・ブログにそのまま貼れます。受け取った人はクリックするだけでブラウザで遊べます。もう「ゲーム送ろうとしたらファイルでかすぎ問題」とは無縁です。
💡 更新の方法 ゲームを修正したら、新しい index.html を上書きアップロード(Step 05 の手順)するだけで、同じ URL のまま自動で内容が更新されます。反映には 1〜数分かかります。
⚠️ 公開できないケース ・index.html が無い → Pages は何も表示できません
・リポジトリが Private → Pro プランでないと公開不可
・404 エラーが出る場合は、URL に /index.html を付けて試してみてください

STEP 07 Git のインストールと初期設定
›

コマンドラインで もっと速く

GUI でも基本は OK ですが、Claude Code と組み合わせるなら Git を入れておくと飛躍的に速くなります。

🍎 Mac の場合

多くの場合、すでに入っています。確認してみましょう。ターミナルを開いて:

git --version

バージョン番号(例:git version 2.39.3)が出れば OK。出ない場合は次の手順で。

未インストールなら:

xcode-select --install

↑ ポップアップが出るので「インストール」をクリック。10 分ほどで完了。

🪟 Windows の場合

  1. git-scm.com/download/win にアクセス
  2. 自動ダウンロードされる .exe をダブルクリックして実行
  3. インストーラーは 「Next」連打で OK(デフォルト設定で問題なし)
  4. 完了後、PowerShell または Git Bash を開いて git --version で確認

👤 初期設定(必須・1 回だけ)

Git に「あなたが誰か」を教えます。これをやらないとコミットできません。

git config --global user.name "あなたの名前"
git config --global user.email "your-email@example.com"

💡 メアドは GitHub に登録したものと同じものを使ってください(コミットがあなたのものとして認識されます)。

🔑 認証の設定(初回プッシュ時)

2021 年からパスワード認証は廃止されました。Personal Access Token(PAT)を使います。

  1. GitHub にログイン → 右上アイコン → Settings
  2. 左メニュー一番下「Developer settings」
  3. Personal access tokens → Tokens (classic) → Generate new token
  4. Note(用途メモ)を入力、Expiration(有効期限)を 90 日 などに設定
  5. 「repo」にチェックを入れて Generate
  6. 表示されたトークンを 必ずコピー(再表示されません!)

💡 初回プッシュ時にユーザー名とパスワードを聞かれたら、パスワード欄にこのトークンを貼り付けます。

✅ ここまで完了したら Git の準備は全部終わり。次のステップで実際にコマンドを使ってみましょう。
08
STEP 08 基本コマンド 7 個でほぼ全部できる
›

毎日使う 必須コマンド

これだけ覚えれば、95% の作業はできます。

1️⃣ git clone(リポジトリをローカルに持ってくる)

git clone https://github.com/ユーザー名/リポジトリ名.git

GitHub 上のリポジトリをパソコンにダウンロード。リポジトリの「Code」緑ボタン → URL コピーで取得できます。

2️⃣ git status(何が変わったか見る)

git status

困ったらまず status。今どんな状態か(未追跡ファイル・変更されたファイル)を表示してくれます。

3️⃣ git add(変更を「次のコミット候補」に追加)

git add .

.(ドット)は「全部」の意味。特定ファイルだけなら git add index.html のように指定できます。

4️⃣ git commit(スナップショット作成)

git commit -m "敵キャラを追加"

-m の後にメッセージ。何を変更したかを一言で書きます(英語でも日本語でも OK)。

5️⃣ git push(GitHub に送信)

git push

これで GitHub のリポジトリに反映されます。初回は git push -u origin main と書きます。

6️⃣ git pull(GitHub から最新を取得)

git pull

別の PC で続きを書く時や、共同作業相手の更新を取り込む時に使います。

7️⃣ git log(履歴を見る)

git log --oneline

過去のコミット履歴を一覧表示。「あの時の状態に戻したい」時のための一覧表。q で抜けます。

📝 典型的な日常ワークフロー

# 1. 何が変わったか確認
git status

# 2. 全変更をステージに上げる
git add .

# 3. メッセージ付きでコミット
git commit -m "ジャンプ機能を追加"

# 4. GitHub に送信
git push

💡 この 4 行をひと続きに覚えれば、開発のサイクルが回ります。

⚠️ よくあるトラブル ・fatal: not a git repository → 別のフォルダに cd で移動するか、git init でリポジトリ初期化
・Permission denied → 認証設定が間違っている。Personal Access Token を再確認
・rejected — non-fast-forward → 先に git pull してから push し直す

STEP 09 Claude Code と連携する
›

Claude Code から 直接 GitHub へ

Claude Code は git コマンドを直接実行できるので、自然言語で頼むだけで全部やってくれます。

🤝 Claude にお願いするだけ

Claude Code を起動した状態で、こう言うだけ:

「このゲームを GitHub にプッシュして。コミットメッセージは『ジャンプ機能を追加』で」

Claude が git status → git add → git commit → git push を順番に実行してくれます。

🆕 新規プロジェクトを GitHub に上げる流れ

ローカルで作り始めたプロジェクトを GitHub にも保管したい時:

  1. GitHub の Web 画面で 空のリポジトリを作る(README にチェックは 入れない)
  2. 表示される指示の 「…or push an existing repository from the command line」のコマンドをコピー
  3. そのままターミナルに貼って実行

💡 Claude に「このプロジェクトを GitHub の新しいリポジトリにアップロードしたい。手順を教えて」と聞けば、現在のディレクトリ構成に合わせた手順を教えてくれます。

🔧 GitHub CLI(gh)を使うとさらに便利

cli.github.com から GitHub CLI(gh コマンド)をインストールすると、ターミナルからリポジトリ作成や Pages 有効化まで全部できるようになります。

gh repo create my-game --public --source=. --push

↑ このコマンド 1 つで、リポジトリ作成 + アップロード + push まで完了!

🚫 .gitignore で不要なファイルを除外

node_modules や .env など、GitHub に上げたくないファイルを .gitignore に書いておきます。

# .gitignore の例
node_modules/
.env
.DS_Store
*.log
dist/
build/

💡 Claude に「HTML/JS ゲーム用の .gitignore を作って」と頼めば、適切なものを生成してくれます。

✅ ここまでマスターすれば Claude Code でゲームを作る → GitHub に push → GitHub Pages で公開 → URL を友達に共有、というサイクルが完成。これは現代のインディー開発者の標準フローです。

💎 GitHub Pages で公開できるもの 12 選 ›

ゲーム以外も 無料公開できます

GitHub Pages は HTML/CSS/JS で動くものなら何でも公開できます。Claude Code で作って、ここに置きましょう。

🎮 HTML/JS ゲーム

本命用途。テトリス・パズル・シューティング・RPG なんでも。Canvas や Phaser.js も動きます。

canvas で動くテトリスゲームを作って。タッチ操作対応、スコア表示、音効果つき。

🌐 ポートフォリオサイト

あなたの実績・プロフィール紹介。就活や案件獲得に有利。

クリエイター向けポートフォリオサイトを作って。トップ・実績・プロフィール・連絡先の4ページ、ダーク&シックな雰囲気。

📝 ブログ

Markdown で書ける Jekyll や Hugo を使えば、シンプルなブログが作れます。

静的サイト生成で動く軽量なブログサイトのテンプレを作って。記事一覧・個別記事・タグ機能つき。

📊 スライド(プレゼン)

reveal.js を使うと PowerPoint いらず。URL を共有して画面共有不要に。

reveal.js を使った10枚のプレゼン資料を作って。テーマは『AI時代のゲーム開発』。

📄 履歴書・職務経歴書

HTML 履歴書はエンジニア界隈で人気。URL を 1 行貼るだけで送れる。

HTMLで作る職務経歴書を作って。シンプルで印刷もしやすいレイアウト、印刷時はPDFになる。

📚 ドキュメント・技術メモ

学んだことをまとめた個人ノート。第二の脳として使えます。

技術メモ用のサイトを作って。サイドバー目次つき、ダークモード対応、検索機能付き。

🎉 イベント告知ページ

勉強会・ライブ・結婚式など。ランディングページ風に作ると効果絶大。

結婚式の招待ページを作って。日時・場所・地図・出欠フォーム・写真ギャラリー付き、上品な雰囲気。

📅 カレンダー・スケジューラ

個人タスク管理ツール。シンプルなものを自分用に。

個人用のカレンダーアプリを作って。月表示、予定の追加・削除、ローカルストレージ保存。

🧮 計算ツール

家計簿・BMI・住宅ローン・税金計算など。「自分が困っているもの」を作るのが最強。

フリーランス用の手取り計算ツールを作って。年収・経費・控除を入力すると手取り月収が出る。

🎨 ジェネレーター

QR コード・ランダム名前・グラデーション色生成など、人気のミニツール。

CSSグラデーションジェネレータを作って。色とアングルを指定するとプレビューと CSS コードがコピーできる。

📈 データダッシュボード

趣味のデータ可視化。読書記録・運動ログ・株価チャートなど。

404 エラーのデバッグ

🔄 4. 日常の開発(毎回使う)

変更を確認してから push

最頻出のフロー。Claude が変更内容を要約してメッセージを提案。

変更を GitHub に push してください。手順:(1) git status と git diff で何が変わったか確認、(2) 変更内容を日本語で要約、(3) 適切なコミットメッセージを 3 案提示、(4) 私が選んだメッセージで git add → commit → push を実行。

機能追加と push をワンセットで

機能名を伝えるだけで、Claude が実装 + コミット + push まで完走。

「[機能名:例 ジャンプ機能 / 効果音 / スコア表示]」を追加してください。実装が完了したら、変更を git にコミットしてください。コミットメッセージは「Add: [機能名]」のように英語で短くお願いします。最後に GitHub に push してください。

最近の進捗を振り返る

「今週何してた?」をログから自動で振り返り。次のアクションも提案。

直近 1 週間のコミット履歴を git log --oneline --since="1 week ago" で確認して、何を作ってきたか日本語で振り返ってください。実装した機能のリスト、改善したこと、次にやるべきことの提案も追加してください。

README の自動アップデート

ゲーム本体の進化に合わせて README も最新化。

README.md を更新したいです。現在のゲームのコードを読んで、次の項目を最新の状態に更新してください:(1) ゲーム概要、(2) 操作方法、(3) 実装済み機能のリスト、(4) 技術スタック、(5) スクリーンショット用プレースホルダー、(6) 開発状況の進捗表示。最後にコミットして push までやってください。

変更を一時退避(stash)

「途中だけど別の作業に切り替えたい」時の退避テク。

今やっている変更をコミットせずに一時退避(stash)したいです。git stash で退避する手順を実行してください。後で戻したい時のために、stash の名前を「[作業内容の短い説明]」で付けてください。一覧確認・復元コマンドも教えてください。

.gitignore を最適化

不要ファイルが git に追跡されないように整える。

現在の .gitignore を確認して、HTML/JS ゲーム開発に最適な内容に更新してください。OS 依存ファイル(.DS_Store、Thumbs.db)、エディタ設定(.vscode、.idea)、ログ、ビルド成果物、機密情報用ファイルを網羅してください。すでに git に追跡されているけど除外したいファイルがあれば、git rm --cached での除外も実行してください。

🆘 5. トラブル対処(困った時の救命プロンプト)

non-fast-forward エラーの解決

push が rejected された時の安全な対処。

git push したら「rejected — non-fast-forward」エラーが出ました。データを失わないことを最優先で、安全に解決してください。手順:(1) リモートの最新を git fetch、(2) 状況を分析、(3) 私のローカル変更とリモート変更の競合を確認、(4) 安全な統合方法(merge / rebase)を提案、(5) 実行。途中で私の確認が必要な箇所では止まってください。

🚨 機密情報を誤って push してしまった

緊急対応プロンプト。優先順位付きで一気に対処。

緊急事態:間違えて [API キー / パスワード / 機密ファイル名] を含むファイルを GitHub に push してしまいました!優先順位順に対応してください:(1) 該当キーを「即座に無効化・再発行する手順」をまず最初に教える(これが最重要)、(2) 該当ファイルを .gitignore に追加、(3) 最新コミットからファイル削除、(4) 必要なら git filter-repo で履歴からも完全削除、(5) 今後の再発防止策。各ステップで私の確認を待ってください。

直前のコミットを修正

メッセージミスやファイル追加忘れの救済。

直前のコミットを修正したいです。まずは git log と git status で状況を確認してください。次の 2 ケースのどちらかを判断・実行してください:(A) まだ push していない場合 → git commit --amend で安全に修正、(B) すでに push してしまった場合 → 履歴を書き換える影響と、安全な代替案を教えてから実行。修正したい内容は「[メッセージを変更したい / ファイルを追加忘れた / etc]」です。

認証エラーの診断

git push で認証拒否された時の原因究明。

git push で認証エラー(Authentication failed / Permission denied など)が出ます。原因を診断してください:(1) リモート URL が HTTPS か SSH かを確認、(2) HTTPS なら Personal Access Token の設定状況をチェック、(3) Token が期限切れの可能性をチェック、(4) git credential の保存状況を確認。それぞれの場合の解決策を提示・実行してください。

「あの時のバージョン」に戻す

機能追加で壊した時に、特定のコミットまで巻き戻す。

最近の変更で何かが壊れたので、特定のコミット時点に戻したいです。git log --oneline で履歴を表示してください。私が指定するコミットハッシュまで戻す方法として、安全な順に提案してください:(1) git revert で戻すコミットを打ち消す(履歴は残る)、(2) 特定ファイルだけ git checkout で戻す、(3) git reset で履歴ごと巻き戻す(危険)。それぞれの違いと推奨度を教えてから実行してください。

プロジェクトのバックアップ実行

💡 こんな時どうする?(よくある質問) ›

初心者あるあるQ&A

Q. Public と Private、結局どっち?

A. ゲーム公開なら Public。GitHub Pages 無料公開には Public が必要。Private は社内コードや、まだ世に出したくないものに。「コードを真似される」心配は実質ゼロです。

Q. 無料で何ができる?有料は必要?

A. 無料(Free プラン)でほぼ全部できます。Public 無制限 / Private 無制限 / GitHub Pages / Actions 月 2,000 分 / Codespaces 月 60 時間 など。個人開発者には十分すぎる量。

Q. 間違えてアップしたファイルを消したい

A. ファイル削除→commit→push で消せます。GUI なら、ファイルを開いて「ゴミ箱アイコン」→ Commit。CLI なら git rm ファイル名 → git commit → git push。ただし履歴には残ります。完全消去には git filter-repo などが必要。

Q. 機密情報(API キー等)を間違えて push してしまった!

A. 即座に対処を:

  1. そのキーを即座に無効化・再発行(履歴から消しても、すでに見られている可能性があるため)
  2. .gitignore に該当ファイルを追加
  3. 必要なら git history から削除(git filter-repo)

⚠️ 順番が大事:まずキーを無効化、それから履歴削除。

Q. GitHub Pages のページが 404 になる

A. 多くの場合、以下の理由:

  • index.html がリポジトリのルートに無い → 移動する
  • Settings → Pages がまだ有効化されていない → 設定を確認
  • デプロイ完了まで 1〜10 分待っていない → 待つ
  • URL の最後に / が抜けている → 付け足す

Q. README.md って何書くの?

A. 5 つの基本要素:

  1. ゲーム名 + 1 行で何のゲームか
  2. 遊べる URL(GitHub Pages のリンク)
  3. スクリーンショットや GIF
  4. 操作方法
  5. 使った技術(HTML/CSS/JS など)

💡 Claude に「このゲーム用の README を書いて」と頼むと、リポジトリ内のコードを読んで適切なものを生成してくれます。

Q. ユーザー名を変更したい

A. Settings → Account → Change username で可能。ただし URL も変わるため、外部リンクが切れます。慎重に。GitHub Pages の URL も 新ユーザー名.github.io/repo に変わります。

Q. 友達と共同開発したい

A. リポジトリの Settings → Collaborators から招待。友達の GitHub ユーザー名を入力すれば一緒に編集可能に。OSS 公開なら誰でも Pull Request(変更提案)できます。

Q. パスワードを忘れた

A. ログイン画面の「Forgot password?」からメールで再設定。Personal Access Token を使っている場合は、それも再発行が必要です。

Q. GitHub と Claude Code はどう違う?

A. 別物です。Claude Code はコードを書くツール(AI アシスタント)、GitHub はコードを保管するサービス。役割分担は:

  • Claude Code → ゲームを作る
  • GitHub → 作ったゲームを保管・公開

💡 両方を組み合わせると、AI 時代のゲーム開発フローが完成します。
